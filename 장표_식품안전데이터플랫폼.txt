【 장표 3 】 분석 추진 흐름도
================================================================================

■ 전체 추진 단계

  ┌──────────────────────────────────────────────────────────────────┐
  │  STEP 1        STEP 2        STEP 3        STEP 4        STEP 5  │
  │  공공데이터    로컬 DB       관계 분석     시각화        플랫폼  │
  │  수집·정제  →  구축        → 자동화      → 구현        → 완성  │
  └──────────────────────────────────────────────────────────────────┘

STEP 1 | 공공데이터 수집·정제
  - 식품의약품안전처 Open API 170개 엔드포인트 크롤링
  - 메타데이터(svc_nm, cat, 컬럼 정의) 수집 → crawl_cache.json 저장
  - 데이터 품질 확인 및 중복·이상값 1차 정제

STEP 2 | 로컬 SQLite DB 구축
  - import-to-sqlite.js 로 170개 테이블 적재
  - api_tables / api_columns 메타 테이블 생성
  - 전체 컬럼 수 2,060개, 레코드 수 확인 및 통계 집계

STEP 3 | PK/FK 관계 분석 자동화
  - analyze_pk_fk.js 스크립트 개발
  - 필드명 유사도 · 값 포함률(inclusion ratio) · 도메인 규칙 복합 스코어링
  - 202개 관계 후보 도출 → HIGH 신뢰도 151개, 실값 교집합 확인 50개

STEP 4 | 데이터 시각화 구현
  - React Flow(직각 ERD) / Vis.js(물리 곡선 네트워크) 이중 구성
  - 키워드 검색 → 관련 테이블 하이라이트
  - SQL 플레이그라운드, 시나리오 추천, 컬럼 탐색 기능 구현

STEP 5 | 플랫폼 완성
  - Node.js Express 서버 + SQLite 읽기전용 연결 분리
  - XSS 방어, Rate Limiting, CORS 설정 등 보안 강화
  - 라우터 모듈화(routes/query, search, datasets, openapi 등)

  식품안전처 Open API를 개별 호출하지 않고 로컬 SQLite에 통합 적재하여

  역할       내용
  ─────────────────────────────────────────────────────────────────
  API 호출   식품안전처 Open API 170개 순차 호출 (배치 처리)
  테이블 생성 CREATE TABLE IF NOT EXISTS — 필드 타입 자동 매핑
  데이터 적재 INSERT OR REPLACE 방식 — 중복 레코드 대체
  메타 저장  api_tables : 테이블 목록·명칭·분류
             api_columns : 컬럼 목록·한글명·SQL 타입

【 장표 6 】 PK/FK 후보 및 테이블 연관성 분석
================================================================================

■ 분석 목적

  수동 매핑 없이 자동으로 테이블 간 조인 가능 관계를 발굴하여
  데이터 융합 활용 경로를 체계적으로 도출함

■ 분석 스크립트 : analyze_pk_fk.js

  분석 로직                내용
  ──────────────────────────────────────────────────────────────────
  ① 필드명 유사도          동일 필드명을 공유하는 테이블 쌍 식별
  ② 도메인 규칙 적용       LCNS_NO, PRDLST_REPORT_NO 등 업무 식별자 우선 처리
  ③ 값 포함률 검증         FROM 테이블 고유값 중 TO 테이블에 존재하는 비율 계산
  ④ 집계 필드 제외         _NM, _DT, _CNT 등 의미 없는 조인 키 필터링
  ⑤ 복합 스코어링          필드 유사도 × 값 포함률 × 도메인 가중치 → 신뢰도 등급

■ 신뢰도 등급 정의

  등급      기준
  ─────────────────────────────────────────────────────
  HIGH      도메인 식별자 + 값 포함률 ≥ 30% 또는 필드 완전 일치
  MEDIUM    필드명 유사 + 값 포함률 ≥ 10%
  LOW       필드명 유사하나 값 교집합 미확인

■ 분석 결과

  항목                    수치
  ──────────────────────────────────────
  전체 관계 후보          202개
  HIGH 신뢰도             151개   (74.8%)
  실값 교집합 확인        50개    (24.8%)
  연관 테이블 수          52개
  핵심 공통 조인 키       LCNS_NO (25개), PRDLST_REPORT_NO (7개),
                          TESTITM_CD (6개), PRDLST_CD (3개)

■ 분석 결과 파일

  foodsafety_key_candidates.json
  ├── relationships[]  : 202개 관계 상세 (from/to 테이블, 키, 신뢰도, inclusion_check)
  └── 각 관계 항목 : from_table, from_field, to_table, confidence, score, reason

■ 키워드 검색 기능

  - 입력어 → AND/OR 조건 선택 가능
  - 전체 테이블·컬럼 값 Full-text 검색 (SQLite LIKE 기반)
  - 검색 결과 : 매칭 테이블 목록 + 매칭 컬럼 · 건수 표시
  - Vis.js 키워드 데이터맵 : 검색어 중심 노드 + 연관 테이블 방사형 배치

  - 카테고리·기관 체크박스 동시 선택 → ERD 필터 연동
  - 선택 없음 = 전체 표시 (null 발행)
  - 선택 있음 = 해당 테이블만 ERD에 표시

■ 트리맵 구성

  - 카테고리별 크기 비례 시각화
  - 클릭 → 해당 카테고리 상세 목록 전환
  - 데이터셋 카드 : 명칭, 테이블 ID, 행 수, 컬럼 수 표시

【 장표 8 】 화면 시안 (Ⅱ) — ERD · SQL 플레이그라운드
================================================================================

■ ERD 화면 구성 (2종)

  [React Flow — 직각 ERD]
  - 정적 스프링 레이아웃 초기 배치 후 물리 시뮬레이션 정제
  - 노드 : 카테고리 색상 카드 (테이블명 + ID + 행 수)
  - 엣지 : 조인 키별 색상 구분, HIGH/MEDIUM/LOW 두께 차등
  - 카테고리 그리드 기반 배치 → 도메인별 클러스터링

  [Vis.js — 물리 곡선 네트워크]
  - SVG 카드 노드 + 곡선 점선 엣지
  - Barnes-Hut 물리 엔진 : 연결 많은 노드가 중앙으로 집결
  - 관계선 : 신뢰도 HIGH → 3px, 그 외 → 2px, 반투명 처리

■ 상세 정보 패널 (우측)

  클릭한 노드 기준으로 우측 패널에 표시
  - 컬럼 명세 (SCHEMA) : 컬럼명 / 타입 / 한글명 / KEY 배지
  - 데이터 샘플 (상위 30행) : 인라인 스크롤 테이블
  - 데이터세트 자세히 보기 → 메인 탐색 패널 이동

■ SQL 플레이그라운드

  - Monaco 스타일 편집기 (코드 하이라이팅)
  - SELECT / EXPLAIN / PRAGMA 전용 화이트리스트 (보안)
  - 결과 : 인라인 테이블 출력 + CSV 다운로드
  - 추천 JOIN 시나리오 자동 삽입 버튼
  - SUPER 등급 3차·5차 체인 JOIN 예시 제공

■ 시나리오 추천 탭

  - 분석 목적 입력 → AI 기반 JOIN 경로 추천
  - 입증된 교집합 관계만 활용하여 쿼리 자동 생성
  - 난이도(easy/medium/hard) 표시 및 필터링

■ 대표 조인 경로 (실증된 5차 체인)

  시험항목(I2530)
    → 기준규격(I0940)   [TESTITM_CD]  포함률 53.4%
    → 위해회수(I0490)   [PRDLST_CD]
    → 수입통관(C001)    [LCNS_NO]
    → 행정처분(I1260)   [LCNS_NO]

■ 관계 유형 분포

  유형    의미                              비율
  ───────────────────────────────────────────────
  1:N     마스터 → 상세 (인허가 → 현황 등) 다수
  N:M     공통 코드 상호 참조               일부
  1:1     동일 업소 복수 등록 테이블        소수

■ 활용 한계 및 유의사항

  - 실 데이터 미적재(0건) 테이블 다수 존재 → 포함률 계산 불가
  - 집계성 필드(_NM, _DT, _CN 등)는 조인 키에서 자동 제외
  - 인허가번호(LCNS_NO) 형식 이원화 → 정규화 전처리 필요

■ 시나리오 분류 체계

  난이도    기준                           대상
  ──────────────────────────────────────────────────

■ 대표 활용 시나리오

  [시나리오 1 — SUPER] 식품 제조사 전체 추적
    I2500(업체인허가) ↔ I1250(품목제조보고) ↔ C002(원재료 성분)
    → 특정 제조사의 전 제품 원재료 구성 분석
    → 응용: 특정 성분 함유 제품 제조사 전수 식별

  [시나리오 2 — SUPER] 위생 위반 업체 실시간 추적
    I0470(행정처분통보) ↔ I0482(행정처분상세) ↔ I2500(업체인허가)
    → 위반 업체의 현재 주소·연락처·영업 상태 일괄 조회
    → 응용: 행정처분 이후 재개업 여부 모니터링

  [시나리오 3 — SUPER] 5차 체인 통합 분석
    I2530(시험항목) → I0940(기준규격) → I0490(위해회수)
    → C001(수입통관) → I1260(행정처분)
    → 시험 항목부터 최종 행정처분까지 전 이력 추적

  [시나리오 4 — MEDIUM] 건강기능식품 인허가 현황
    I1290(건강기능식품 판매업허가) ↔ I2822(건강기능식품 폐업정보)
    → 영업 중 업체 vs 폐업 업체 비교 현황판

  [시나리오 5 — EASY] 수입식품 이력 단순 조회
    C001(수입통관) 단독
    → 특정 국가·품목 수입 이력 기간별 조회

  - 데이터맵 ERD에서 노드 클릭 → 관련 시나리오 바로가기
